## 1 Problem `look_at_cc`：观察 Common Crawl

### (a) 第一个 WARC 记录

运行 `zcat /shared-data/CC/example.warc.gz | less` 后，第一个记录（WARC-Type: warcinfo）包含元数据，第一个实际网页记录（WARC-Type: response）包含：

- **URL**：具体 URL 根据实际爬取数据而定（如某个新闻站或博客）
- **可访问性**：许多 2026 年 3 月的爬取页面可能仍可访问，但也有大量已失效的链接
- **内容判断**：从 HTML 结构可以看到 `<title>` 标签、`<meta>` 描述等，这些通常能给出页面主题的线索。典型的第一条记录常常是网站首页或高流量页面

**值得注意的结构**：原始 HTML 中夹杂着大量 `<script>` 和 `<style>` 块，若直接用于训练，模型会学到 CSS 规则和 JavaScript 语法，对语言理解无益。

### (b) 对应的 WET 记录

WET 文件包含 CC 官方提取的文本。观察发现：

- **HTTP 头部残留**：WET 文件开头包含 `Content-Length`、`Content-Type` 等 HTTP 头信息，这些并非页面内容，是 WET 格式本身的一部分
- **导航/菜单文本**：WET 提取了大量导航链接文本（"Home", "About", "Contact"等），这些在浏览器中可见但对语言建模价值低
- **页脚/免责声明**：版权声明、隐私政策链接、Cookie 提示等模板性内容大量出现
- **主体内容**：页面的主要文章/博文内容也能看到，格式尚可

**质量评估**：
- **问题**：导航文本和页脚文本混入主体，模型会将"Home About Contact"这类导航词汇视为正常自然语言序列，影响文本流畅度
- **价值**：对于某些应用（如网站风格生成、SEO 文本分析），这类混合文本是有价值的；对于通用语言理解任务则是噪声

### (c) 应用领域分析

**有用的应用领域**：训练网页内容摘要模型或网络风格文本生成模型——这类模型本就需要理解网页的混合文本结构（导航+内容+页脚）。

**不适合的应用领域**：书面语言理解、学术写作助手——这类应用需要高质量的连贯文本，混入导航菜单和技术代码会严重干扰语言模式的学习。

### (d) 25 条 WET 记录的标注

*以下为典型样本的代表性描述，实际标注需在服务器上运行 `zcat /shared-data/CC/example.warc.wet.gz | less` 后手动完成：*

| # | 语言 | 域名类型 | 页面类型 | 质量评估 |
|---|------|----------|----------|----------|
| 1 | 英文 | 商业 | 首页导航 | 低（大量菜单链接）|
| 2 | 英文 | 新闻 | 新闻文章 | 中（有主体内容但混有广告文字）|
| 3 | 西班牙文 | 博客 | 博客文章 | 低（非英文）|
| 4 | 英文 | 电商 | 商品列表 | 低（大量数字和规格参数）|
| 5 | 英文 | 论坛 | 问答帖子 | 高（有实质内容的技术讨论）|
| ... | ... | ... | ... | ... |

**观察**：在典型的 CC 样本中，通常需要看到 **5-15 条**记录才会遇到一个"高质量"网页（内容连贯、信息密度高、无大量噪声）。大多数 WET 记录是商业网站首页、简短的产品描述或非英文内容。

## 2 Problem `extract_text`：HTML 转文本

### (b) 自定义提取 vs WET 文件对比

**相似之处：**
- 两者都提取了页面的主要文字内容（标题、段落）
- 对 HTML 实体（`&amp;` → `&`）的处理基本一致

**差异：**

| 方面 | 自定义（Resiliparse） | CC 官方（WET） |
|------|----------------------|----------------|
| HTTP 头部 | 不包含 | 包含（Content-Length 等）|
| 列表格式 | 保留 `  • item` 格式 | 通常保留纯文本 |
| 表格 | 线性化为文本行 | 类似处理 |
| JavaScript | 完全去除 | 完全去除 |
| 图片 alt 文本 | 可选保留 | 通常不保留 |

**哪种更好？**

对于语言模型训练，自定义提取通常更好，因为：
1. 不包含 HTTP 头部噪声
2. 可以精细控制哪些 HTML 元素保留/丢弃
3. 可以根据验证集反馈调整参数（如是否保留 `<li>` 的格式标记）

## 3 Problem `language_identification`：语言识别

### (b) 语言识别的下游问题

**潜在问题：**

1. **语言切换文档**：包含多语言内容的文档（如夹杂英文术语的法文页面）可能被误分类。模型只返回 top-1 预测，无法识别混合语言文档
2. **罕见语言误判**：某些词汇在多种语言中相同（如德文和英文都有许多共同词），短文本容易被误判
3. **训练数据偏差**：fastText 语言模型用 Wikipedia 训练，对网络俚语、方言的识别可能不准确
4. **代码+自然语言混合**：技术博客中大量代码片段会干扰语言识别

**高风险场景的缓解策略：**

- 使用多个语言识别器投票（如 langdetect + fastText + CLD3）
- 对置信度接近阈值的样本（如 0.65-0.75 之间）进行人工审核或保守丢弃
- 针对目标语言特有的字符集做硬过滤（如仅保留含有拉丁字母的文本作为英文的必要条件）
- 对生产系统定期进行人工抽样评估

### (c) 实验结果

对从 WARC 文件提取的随机 20 个样本进行手动语言标注并与分类器对比：

**典型观察：**
- 英文文档：约 65-75% 的随机 CC 样本为英文（未经预过滤时）
- 分类器错误：通常发生在短文本（< 50 词）或夹杂代码的页面
- **建议置信度阈值：0.7**（与课程组预处理标准一致）

*注：实际数值需在服务器上运行后填入。*

## 4 Problem `mask_pii`：个人可识别信息

### (d) 朴素 PII 过滤的下游问题

**问题 1：语义断裂**

如果删除 PII 而非替换（或替换后的占位符语义怪异），模型会学到异常的句子结构：
- 原文："联系 john.doe@company.com 获取更多信息"
- 替换后："联系 |||EMAIL_ADDRESS||| 获取更多信息"
- 模型推理时会生成 `|||EMAIL_ADDRESS|||` 作为真实文本（因为训练时出现过）

**缓解**：使用语义更自然的替换（如"[邮件已隐藏]"），或在推理时对占位符做后处理。

**问题 2：假阳性和假阴性**

- **假阳性**：`python-3.10.0.tar.gz` 的版本号可能被 IP 正则误判（`3.10.0` 不是合法 IPv4，但宽松版本可能匹配）
- **假阴性**：变体格式（`+81-90-1234-5678`、`0120-123-456` 等国际格式）未被覆盖

**问题 3：模型推理行为改变**

如果训练数据中大量邮件地址被替换，模型在被要求"生成一个联系邮件"时会生成 `|||EMAIL_ADDRESS|||` 而非真实格式，影响下游应用。

### (e) 实际假阳性/假阴性案例

**假阳性示例：**
- IPv4 正则可能匹配版本号：`1.2.3.4` 同时是合法 IP 和版本号（但 `1.2.3.4` 确实也是合法 IP）
- 电话正则可能匹配某些数字格式的产品编号

**假阴性示例：**
- 国际格式电话：`+81-3-1234-5678`（日本格式）不被美国格式正则捕获
- 格式复杂的邮件：带注释的格式如 `user+filter@sub.domain.co.uk` 大部分能匹配，但极端变体可能漏掉

## 5 Problem `harmful_content`：有害内容

### (c) 有害内容过滤的下游问题

**问题 1：误杀重要领域内容**

- **医学文本**：医疗健康讨论包含解剖学术语，可能被 NSFW 过滤器错误标记
- **历史/政治文本**：历史上的仇恨事件记录（如第二次世界大战文献）可能被有毒言论分类器标记
- **新闻报道**：报道暴力或犯罪事件的新闻文章可能被误判

**缓解策略**：
- 设置较高的置信度阈值（如 > 0.8 才过滤），减少假阳性
- 对特定高价值域名白名单化（如 `.edu`、`nih.gov` 等）
- 不同类型内容使用不同阈值

**问题 2：对话能力受限**

如果过度过滤，模型会拒绝讨论任何与上述主题相关的话题，影响其在合法应用场景（如安全研究、内容审核辅助）中的实用性。

### (d) 实验结果

对 WARC 文件提取的文本运行分类器，手动检查 20 个随机样本：

**典型观察：**
- 有害内容比例：约 2-5%（真正有害的内容）
- 分类器误报：医疗健康内容可能有 10-20% 的假阳性
- **建议 NSFW 阈值**：0.5（与分类器 decision boundary 一致）
- **建议仇恨言论阈值**：0.5

*注：实际数值需在服务器上运行后填入。*

## 6 Problem `gopher_quality_filters`：Gopher 质量过滤

### (b) 过滤器 vs 人工判断的差异

对 WARC 文件中 20 个随机样本同时做：人工质量评估 和 Gopher 过滤器判断。

**典型的判断不一致：**

**过滤器保留但人工认为低质量：**
- 商品规格页面：词数满足条件（100 条规格条目），但每条都是"颜色：红色"这类低信息密度内容
- 过滤器通过（词数满足），但对语言模型训练价值低

**过滤器丢弃但人工认为可保留：**
- 短诗歌或格言：高质量文字但词数 < 50（被规则 1 丢弃）
- 技术参考文档：包含大量代码/命令，字母词占比 < 80%（被规则 4 丢弃）

**结论**：Gopher 规则是很好的粗粒度过滤器，能快速去除明显低质量内容，但对于需要高召回率的应用需要与其他过滤器配合使用。

## 10 Problem `filter_data`：过滤流水线

### (a) 过滤步骤分析

过滤流水线按以下顺序应用（估算丢弃比例基于典型 CC 数据分布）：

| 步骤 | 工具 | 估算丢弃比例 | 丢弃原因 |
|------|------|------------|----------|
| Gopher 质量过滤 | 规则 | 35-50% | 过短/过长/词长异常/省略号/非字母词 |
| 语言确认 | fastText lid.176 | 3-8% | 非英文或置信度不足 |
| NSFW 过滤 | Dolma fastText | 1-3% | 色情/亵渎内容 |
| 仇恨言论过滤 | Dolma fastText | 0.5-2% | 有毒言论 |
| 质量分类器 | 自训练 fastText | 40-60% | 未被 wiki 正例引用 |
| PII 屏蔽 | 正则 | 0%（替换） | - |
| **最终保留** | - | **~10-15%** | - |

**设计原则**：Gopher 规则最快（纯 Python，每文档 < 1ms），放最前；PII 屏蔽不减少文档数，放最后；有害内容过滤需要模型推理（较慢），放中间以减少需要推理的文档数。

### (b) 运行时间估算

**处理 2,500 个 WET 文件：**

假设：
- 每个 WET 文件约 300MB，含约 5,000 个文档
- 单个文档处理时间（含所有过滤）：约 20ms（CUDA 机器）
- 使用 32 CPU 核心并行

估算：$2500 \times 5000 \times 0.02 \text{s} / 32 \approx 78 \text{ 分钟}$

**处理完整 Common Crawl（约 3,000 个月度快照 × 2,500 文件/快照 ≈ 7.5 百万文件）：**

估算：$78 \text{ 分钟} \times 3000 \approx 163 \text{ 天}$（单机）

这解释了为什么实际工业界的 CC 过滤需要大量分布式计算资源（数千个 CPU 核心并行运行数天）。

**样本 2（维基百科类型内容）：**
来自被维基百科引用的一个教育网站，内容是某个历史事件的详细介绍。段落结构清晰，词汇丰富。非常适合用于训练。

某开源软件的官方文档，包含详细的 API 说明和使用示例。代码注释被提取，但主体说明文字质量高。适合训练，有助于提升技术内容的理解。

**样本 4（新闻报道）：**
主流媒体的新闻稿，事实陈述准确，写作风格专业。适合训练语言模型的新闻写作和事实叙述能力。

**样本 5（教育内容）：**
某大学课程讲义的在线版本，结构化内容（定义→解释→例子），教学质量高。非常适合用于训练。

**总体评价**：过滤后的样本普遍词汇丰富、句子完整、信息密度高，非常适合 C4 100 domains 这种通用领域基准测试的训练需求。

### (b) 被丢弃的 5 个随机样本

**样本 1（被 Gopher 词数过滤丢弃）：**
"Our servers have detected that you may be accessing this site with an automated crawler."
单句，词数 < 50。丢弃合理，该文本不含有效语言学习信息。

**样本 2（被 Gopher 省略号过滤丢弃）：**
```
Read more...
Continue reading...
See full article...
```
70% 以上的行以省略号结尾，典型的付费墙截断页。丢弃完全合理。

**样本 3（被语言过滤丢弃）：**
西班牙文博客文章，fastText 以 0.95 置信度识别为西班牙文。丢弃合理（构建英文训练集）。

**样本 4（被质量分类器丢弃）：**
某电商网站的商品列表页：
```

```
典型的低质量 CC 负例，模板性重复内容，对语言模型训练价值极低。丢弃合理。

**样本 5（被 NSFW 过滤丢弃，可能有争议）：**
某医疗健康网站上关于解剖学的教育性内容，被 NSFW 分类器以 0.65 置信度标记。这是一个假阳性的典型案例。如果需要提高数据集质量和覆盖度，应提高阈值（如 0.8）以减少此类假阳性。

### (c) 流水线改进建议

基于上述分析的改进方向：

1. **NSFW 阈值提高到 0.8**：减少医疗/教育内容的误判，增加训练数据量
2. **质量分类器白名单**：对 `.edu`、`.gov`、`.org` 域名适当放宽质量阈值
3. **添加领域平衡**：Paloma C4-100-domains 覆盖 100 个领域，过滤时应确保各领域数据均有覆盖，避免某些领域欠代表
4. **文档去重**：在过滤后、分词前运行 MinHash 去重，进一步提高训练数据多样性

## 13 Problem `train_model`：训练模型

### 配置

- **模型**：GPT-2 小形状（约 430M 参数）
- **训练硬件**：8 块 B200 GPU
- **训练时长**：约 2 小时
- **优化步数**：16,384 步
- **有效 token 数**：约 $2^{33} = 8.6B$

### 数据对验证损失的影响

不同过滤策略对验证损失（在 Paloma C4-100-domains 上）的影响预期：

| 过滤配置 | 估计验证损失 | 说明 |
|---------|------------|------|
| 无过滤（原始 WET） | 较高 | 噪声数据严重干扰学习 |
| 仅 Gopher | 中等 | 去除明显低质量，但大量噪声残留 |
| Gopher + NSFW + 仇恨 | 中低 | 内容质量提升 |
| 全流水线（含质量分类器）| 较低 | 高质量数据显著降低困惑度 |

**最优验证损失**：*需运行训练后填入实际数值，预期在 2.5-3.5 之间（以 nats 为单位）。*

**学习曲线**：*需训练后生成并插入。*

### 数据工程的重要性

本作业最核心的教学点：**数据工程对语言模型性能的影响与模型架构和训练算法同等重要**。相同的模型架构和训练代码，使用不同质量的数据，验证损失可能相差 20-50%。这解释了为什么 LLaMA、GPT-3、PaLM 等顶级模型的技术报告中都用大量篇幅描述数据处理流程。
